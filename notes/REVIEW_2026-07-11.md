## 🔴 Critique — le serveur tombe entièrement

### [x] 1. N'importe quelle erreur réseau tue **tout le process**, pas juste le client fautif

**Reproduit à 100%.** Un simple reset TCP (RST) envoyé par un client suffit à
faire crasher le serveur pour **tout le monde** :

```python
s = socket.socket(...)
s.connect(("127.0.0.1", 8090))
s.send(b"GET / HTTP/1.1\r\nHost: localhost\r\n")  # requête incomplète
s.setsockopt(socket.SOL_SOCKET, socket.SO_LINGER, struct.pack('ii', 1, 0))
s.close()  # force un RST au lieu d'un close propre
```

Résultat : le process s'arrête net, stderr affiche juste `500`, et le
process n'existe plus (`pgrep` ne le retrouve pas). C'est aussi ce qui a fait
tomber le run k6 (`tests/k6/stress.js`) : 39% d'échecs parce que le serveur
est mort en cours de charge après quelques dizaines de requêtes concurrentes.

**Pourquoi :**
`Client::handleRecv()` (`server/client/Client.cpp:59`) et
`Client::handleSend()` (`server/client/Client.cpp:136`) font
`throw std::runtime_error("500")` dès que `recv()`/`send()` renvoie `< 0`
(ex: `ECONNRESET`). Pareil pour `Server::readCgiPipe()`
(`server/Server.cpp:204`) sur une erreur de `read()`.

Le souci : dans `Server::clientRead()`, seul le bloc `parseRequest(...)`
est dans un `try/catch` — l'appel à `client.handleRecv()` juste avant est
**hors du try**. Donc l'exception remonte : `clientRead` → `loopPollFds` →
    `Server::run` → `main()`, où le seul `catch` restant se contente d'un
`cerr << e.what()` puis `return 0`. **Tout le serveur s'éteint** à cause
d'un seul client.

**Piste :** ces trois `throw` (recv, send, pipe CGI) doivent être catchés
*localement*, au niveau de la connexion concernée, pas remonter jusqu'à
`main`. Le geste correct dans ce cas précis n'est même pas de construire une
réponse d'erreur (le socket est probablement mort) : c'est de fermer *ce*
client (`closeClient`) et continuer la boucle `poll()` pour les autres. Il
faut aussi se demander si `EAGAIN`/`EWOULDBLOCK` devraient vraiment jeter une
exception (cf. #10 : les fds clients ne sont jamais passés en non-bloquant).

### [x] 2. Ctrl+C (SIGINT) ne marche pas — le process ne répond pas au signal

**Reproduit.** `kill -SIGINT <pid>` (2 fois, plusieurs secondes d'attente) ne
fait strictement rien — pas de "SIGNAL RECEIVED" sur stderr, le process reste
`poll()`-bloqué indéfiniment. Il a fallu un `SIGTERM` pour le tuer.

**Pourquoi :** `signal(SIGINT, handle_sigint)` n'est appelé que dans
`Server::Server(std::string name)` (`server/Server.cpp:30-35`). Mais le
`Parser` construit toujours ses serveurs avec le constructeur par défaut :

```cpp
Server new_server;                      // Parser.cpp:112 — Server::Server(void)
_servers_vector.push_back(new_server);
```

`Server::Server(void)` (`Server.cpp:27`) n'appelle jamais `signal()`. Le
constructeur qui l'appelle n'est tout simplement jamais utilisé par le
parsing réel — c'est du code mort qui donne l'illusion que Ctrl+C est géré.

**Piste :** le `signal()` doit s'enregistrer quelque part qui tourne
*toujours*, indépendamment de quel constructeur `Server` a été utilisé pour
construire tel ou tel objet — par exemple une fois dans `main()` avant de
lancer `run()`, ou dans les deux constructeurs (mais ça reste fragile si un
3ᵉ constructeur apparaît). C'est un vrai problème pour le rendu/l'éval : un
process qu'on ne peut pas arrêter proprement, il faut le `kill -9`.

### [x] 3. `pollfd.revents` non initialisé — confirmé par Valgrind

```
==Memcheck== Conditional jump or move depends on uninitialised value(s)
   at Server::loopPollFds() (Server.cpp:333)
   by Server::run(...) (Server.cpp:404)
Uninitialised value was created by a stack allocation
   at Server::handleCgi(Client&, int) (Server.cpp:237)
```

**Pourquoi :** dans `Server::handleCgi()` (`Server.cpp:240-243`) :

```cpp
pollfd pfd;
pfd.fd = client.getCgiPipefd();
pfd.events = POLLIN;
_poll_fds.push_back(pfd);   // pfd.revents n'est JAMAIS initialisé
```

`loopPollFds()` boucle avec `for (size_t i = 0; i < _poll_fds.size(); i++)`
et relit `.size()` à chaque tour — donc si un CGI est lancé pendant cette
même itération de boucle (le pipe est ajouté à un index > `i` courant), la
boucle va l'atteindre **dans le même appel**, avant que `poll()` n'ait
jamais eu la chance de remplir son `.revents`. Le test `if (!_poll_fds[i].revents) continue;`
lit alors de la mémoire de pile non initialisée : selon ce qu'il y avait sur
la pile, le pipe fraîchement créé peut être traité comme "prêt" à tort (et
potentiellement faire partir un `accept()`/`read()` bloquant sur le mauvais
fd), ou au contraire être ignoré alors qu'il l'est vraiment.

Le second site `pollfd pfd;` (`Server.cpp:393`, dans `run()` pour les
listeners) est correct : il fait bien `pfd.revents = 0;` juste après.

**Piste :** initialise `revents` (à `0`) partout où un `pollfd` est construit
à la main, pas seulement à un des deux endroits.

## 🟠 Sévère

### [x] 4. `_counter_trash` déborde (unsigned underflow) → le 413 ne part jamais, la connexion reste bloquée

C'est la logique que ton propre `tests/k6/stress.js` cible déjà
(`POST /uploads (oversized > 50M)` → attend un `413`). En solo (une requête
curl à la fois) ça passe, mais sous charge concurrente le taux d'échec sur ce
check tombe à ~59%.

**Pourquoi :** `_counter_trash` est un `size_t` (non signé)
(`server/client/Client.hpp:33`). Dans `Client::handleRecv()` :

```cpp
if (_status == TRASH) {
  _counter_trash -= n;   // Client.cpp:64
  return true;
}
```

`n` (octets réellement reçus par ce `recv()`, jusqu'à `STD_BUFFER`=4096) peut
tout à fait dépasser ce qu'il reste à purger (ex: il reste 10 octets à
jeter, mais le buffer noyau en contient 4096 de prêts). `_counter_trash -= n`
avec des types non signés **ne devient pas négatif : il boucle vers
`SIZE_MAX`**. Ensuite, dans `Server::clientRead()` :

```cpp
if (client.getCounterTrash() <= 0 && client.getStatus() == TRASH) { ... }
```

`<= 0` sur un `size_t` ne redevient jamais vrai après l'overflow (seul `== 0`
compte). Le 413 ne se construit jamais, le fd reste en lecture pour
toujours : c'est une fuite de connexion (et, à l'échelle, un moyen simple de
faire tenir un paquet de sockets ouverts indéfiniment côté serveur).

**Piste :** le calcul doit détecter quand `n` dépasse ce qu'il reste à
purger et clamp à `0` plutôt que de soustraire aveuglément — c'est le même
genre de garde qu'on met pour éviter un `size() - x` qui sous-flow ailleurs
dans le projet (t'as déjà ce réflexe vu `Request.cpp`/`StaticHandler.cpp`).
Vérifie aussi les deux blocs de `isRequestCompleted()`
(`Client.cpp:91-99`, Content-Length) qui font le même calcul en double —
factorisable, et regarde s'il ne faut pas la même garde côté chunked
(`Client.cpp:106`, où `_counter_trash` n'est même pas mis à jour quand on
passe en `TRASH`).

### [x] 5. Le body d'une requête chunked n'atteint jamais le script CGI

**Reproduit.** Deux POST JSON identiques vers `/cgi-bin/serve.py`, un en
`Content-Length` normal, un forcé en `Transfer-Encoding: chunked` :

```
Content-Length normal  → HTTP/1.1 201 Created
Transfer-Encoding: chunked → HTTP/1.1 400 Bad Request
```

Le script CGI reçoit un body vide dans le cas chunked, alors que
`Request::decodeChunk()` a bien décodé le body côté C++.

**Pourquoi :** dans `Request::parseRequest()` (`Request.cpp:119`), côté
chunked :

```cpp
headers["Content-length"] = body.size();
```

Deux problèmes empilés ici :
- La clé est `"Content-length"` (petit `l`), alors que partout ailleurs
  (`Request.cpp:108`, `113`, `163`) le code cherche `"Content-Length"`
  (grand `L`). `std::map<std::string,...>` est sensible à la casse : la clé
  chunked n'est **jamais retrouvée** par `parseCgi_env()`, qui met donc
  toujours `CONTENT_LENGTH=0` dans l'env du CGI pour une requête chunked.
- Même en supposant la casse corrigée : `body.size()` est un `size_t`, et
  `std::string` n'a pas de constructeur/`operator=` prenant un entier — ce
  qui compile quand même (`-Wall -Wextra -Werror -std=c++98` ne dit rien)
  parce que `size_t` se convertit implicitement en `char` via
  `operator=(char)`. Le header ne contiendrait donc pas "123" mais **un seul
  caractère tronqué**.

Conséquence en cascade : `Cgi::getContentLength()` lit `CONTENT_LENGTH=0`,
donc `Cgi::dadaExec()` (`Cgi.cpp:107`, `if (content_length > 0)`) n'écrit
jamais le body décodé dans le pipe vers le script. Toute requête chunked
vers un CGI (upload via chunked, par ex.) perd silencieusement son body.

**Piste :** il faut mettre à jour `headers["Content-Length"]` (bonne casse)
avec une vraie conversion entier→string (regarde comment le reste du code
convertit déjà des tailles en string ailleurs, ex: `Response::build()`).

### [x] 6. `close(pipe_body[2])` — accès hors-limites dans `pipe_and_fork`

`server/cgi/Cgi.cpp:73` :

```cpp
if (pipe(pipe_resp) == -1) {
    close(pipe_body[1]);
    close(pipe_body[2]);   // <- pipe_body est un int[2] ! index 2 = hors tableau
    ...
}
```

`pipe_body` est déclaré `int pipe_body[2]` (`Cgi.cpp:127`). `pipe_body[2]`
lit/ferme un fd au hasard dans la pile — comportement indéfini, potentiel
`close()` d'un fd valide appartenant à autre chose (une autre socket
cliente, un autre pipe...). Ce n'est déclenché que si le **second** `pipe()`
échoue (donc rare en usage normal — ça demande d'épuiser les fds
disponibles), mais c'est un bug net, pas une supposition.

**Piste :** relis les deux fds ouverts par le premier `pipe()` réussi
(`pipe_body[0]` et `pipe_body[1]`) — lequel manque à l'appel ?

### [ ] 7. Le routing multi-serveur (virtual host) ignore complètement le `Host` / le listener touché

**Reproduit.** `webserv.conf` a deux blocs `server` qui partagent le port
`8110` (`localhost` et `tchoutchou`), avec des redirections `/old`
différentes (`/files` vs `/new`). Peu importe le `Host:` envoyé :

```
curl -H "Host: tchoutchou" http://127.0.0.1:8110/old → Location: /files
curl -H "Host: localhost"  http://127.0.0.1:8110/old → Location: /files
```

`tchoutchou` devrait donner `/new` — il donne toujours `/files` (config du
1ᵉʳ serveur).

**Pourquoi :** `main.cpp:49` ne lance que
`servers_vector[0].run(listeners)`. `Server::run()` ajoute bien tous les fds
de tous les listeners au `poll()` (donc les deux ports/tous les vhosts
acceptent des connexions), mais `Server::clientRead()`/`handleReq()`
utilisent toujours `_locations_vector` — qui appartient à **ce** `Server`
(le n°0), quel que soit le listener/le port/le Host réel de la requête.
C'est cohérent avec ce qui était déjà noté dans `notes/REVIEW.md` (juin) :
"Host-based server selection" pas câblé — ça reste vrai aujourd'hui.

**Piste :** il manque le lien `listen_fd → Listener*` (pour savoir sur quel
listener/port la connexion est arrivée), puis `Listener → quels Server*`
(déjà présent via `getLinkedServers()`), puis matcher le header `Host` parmi
ces serveurs candidats. Cette classe `Client` devra porter une référence vers
"son" `Server` (ou ses candidats) plutôt que ce soit toujours celui qui
tourne `run()`.

## 🟡 Moyen

### [ ] 8. Écriture du body CGI 100% bloquante

`Cgi::dadaExec()` (`Cgi.cpp:108`) fait un seul `write()` bloquant de tout le
body vers le pipe du script, avant même de retourner dans la boucle
`poll()`. Un body plus gros que le buffer du pipe (~64 Ko sous Linux) et un
script qui ne lit pas assez vite bloque **tout le event loop** (donc tous
les autres clients) le temps que ça se débloque. Déjà signalé dans l'ancien
`notes/REVIEW.md`, toujours vrai. Cette écriture doit passer par le `poll()`
comme la lecture du pipe de sortie (POLLOUT sur le fd d'écriture, écriture
partielle, etc.) plutôt qu'un `write()` synchrone.

### [ ] 9. `_pipe_to_client` grossit indéfiniment (fuite mémoire lente)

`Server::readCgiPipe()` (`Server.cpp:213`), sur EOF du pipe CGI, fait
`closeClient(i, fd)` avec `fd` = **le fd du pipe** (pas celui du client).
`closeClient()` (`Server.cpp:226-230`) fait `close(fd)` (ok, ferme bien le
pipe) et `_clients.erase(fd)` — qui ne fait rien puisque `_clients` est
indexé par fd *client*, jamais par fd de pipe (ça marche "par accident" —
`erase()` sur une clé absente est un no-op silencieux, pas un crash). Mais
**l'entrée correspondante dans `_pipe_to_client[fd] = client_fd` n'est
jamais retirée** (`Server.hpp:26`). Cette map grossit d'une entrée à chaque
requête CGI traitée, sans jamais rétrécir, pour toute la durée de vie du
process — fuite mémoire lente mais réelle sur un serveur long-running.

**Piste :** `readCgiPipe` doit faire `_pipe_to_client.erase(fd)` en plus du
`close()`. Vaut aussi le coup de splitter le rôle de `closeClient` (voir
#13) pour que ce genre de confusion (client fd vs pipe fd) n'arrive plus.

### [ ] 10. `Client::~Client()` ne ferme aucun fd — tout repose sur `closeClient()` explicite

`server/client/Client.cpp:51` : `Client::~Client(void) {}` — vide. Ni
`_poll_listen.fd` ni `_cgi_pipe_fd` ne sont jamais fermés par le destructeur.
Ça "marche" tant que tous les chemins de sortie appellent bien
`closeClient()` — mais on vient de voir (#1) que les chemins d'exception ne
le font pas, et qu'un client peut disparaître de `_clients` (erasé par
erreur, cf #9) sans que son fd soit fermé. Aussi : aucun fd client
accepté n'est jamais passé en `O_NONBLOCK` (`acceptNewClient`,
`Server.cpp:219-224`, ne fait que `FD_CLOEXEC`) — sur un serveur
mono-thread piloté par `poll()`, c'est un choix qui peut *sembler* marcher
tant que les clients sont bien élevés, mais qui n'est pas correct par
construction.

### [x] 11. Dépassement potentiel de `_max_body_size` via un `Content-Length` énorme

`utils/utils.cpp:11-16`, `strToInt()` fait `stringstream >> size_t`. Un
`Content-Length` volontairement énorme (plus grand que ce qu'un `size_t`
peut représenter proprement une fois reconverti) peut, selon la valeur,
donner un résultat surprenant une fois castée en `long` dans
`isRequestCompleted()` (`Client.cpp:91`, `static_cast<long>(len)`). Pas
reproduit à l'exécution (pas eu le temps de forcer un dépassement propre),
mais mérite un test dédié avec un header `Content-Length: 99999999999999999999`
avant le rendu — potentiel contournement de la limite de taille.

## 🟢 Mineur / nettoyage

- **[ ]** `Server::acceptNewClient(int listen_fd)` (`Server.cpp:219`) : le
  paramètre s'appelle `listen_fd` mais reçoit en réalité le fd **accepté**
  (`acceptNewClient(client_fd)` à l'appel, `Server.cpp:358`). Le
  `fcntl(listen_fd, F_SETFD, FD_CLOEXEC)` s'applique donc bien au socket
  client, pas au listener — comportement correct, mais le nom trompe (t'as
  dû t'y reprendre à deux fois en le relisant, non ?).
- **[ ]** `closeClient()` est détourné pour fermer un fd de pipe CGI (voir
  #9) — deux responsabilités différentes (fermer un client vs fermer un
  pipe) dans une fonction au nom qui n'en dit qu'une.
- **[ ]** `server/Location.hpp:56` : `Location* findLocation();` — déclaration
  sans définition, sans paramètres, jamais appelée (le vrai

- **[ ]** Racine du repo : `vgcore.769653`, `vgcore.771830` (36 Mo chacun,
  09h35/09h40 aujourd'hui) — des core dumps d'une session de debug
  précédente, à supprimer ou `.gitignore`r si ce n'est pas déjà fait (je n'y
  ai pas touché, à toi de voir si t'en as encore besoin).

## Ce qui va bien (pas de régression trouvée)

- **Pas de zombie** : 30 requêtes CGI concurrentes → aucun process enfant
  restant, `waitpid(-1, NULL, WNOHANG)` en fin de `loopPollFds()` fait le
  travail correctement sur le chemin heureux.
- **Pas de fd leak sur le chemin heureux** : 100 GET + 50 GET CGI
  séquentiels → fd count du process stable (6 avant/après).
- **Valgrind, chemin heureux (GET, CGI GET/POST, upload multipart, DELETE,
  autoindex, 405, 413)** : `definitely lost: 0`, `indirectly lost: 0`,
  `possibly lost: 0`. Seul `still reachable: 108 757 bytes` en sortie — mais
  le process a dû être tué au `SIGTERM` (à cause du bug #2, impossible de le
  stopper proprement au `SIGINT`), donc les destructeurs de `main` n'ont
  jamais tourné : ce chiffre ne veut rien dire tant que #2 n'est pas
  corrigé. À re-tester une fois #2 fixé, avec un vrai `SIGINT` propre, pour
  avoir un résumé Valgrind qui reflète un arrêt normal.
- Parsing de config (`Lexer`/`Parser`/`Token`) : logique saine, bonnes
  validations (host, port, méthodes, tailles avec suffixes K/M/G, codes
  d'erreur bornés 300-599). Rien trouvé à redire dessus.
- Build clean : `make fclean && make` ne sort **aucun warning** avec
  `-Wall -Wextra -Werror -Wpedantic -Wshadow -Wold-style-cast`.
