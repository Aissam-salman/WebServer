# WebServ — Revue complète (11 → 12 juillet 2026)

Fusion des deux revues en un seul document : la revue statique/dynamique du
11/07 (sur l'ancienne arborescence `server/`) et sa mise à jour du 12/07
après le merge `TimPart4`+`TimPart5` → `main` → `handleOver` (commit
`ccbff42`), qui a déplacé tout le code vers `src/{core,http,cgi,client,config,utils}/`
et sorti la boucle événementielle de `Server` vers une nouvelle classe
`WebServ` (`src/core/WebServ.cpp`).

Chaque point a une case à cocher : coche-la une fois corrigée et
revérifiée. Les chemins de fichiers indiqués sont ceux **après** le merge
(`src/...`) sauf mention contraire explicite ("avant le merge").

**But de ce doc :** un diagnostic précis (fichier:ligne, pourquoi c'est
cassé, comment le reproduire) pour aller directement au fix, sans
re-débugger depuis zéro. La direction du fix est donnée, pas le patch tout
cuit — surtout pour les points qui touchent à l'architecture (#1, #7, #8,
#9, #12) et méritent d'être pensés par toi.

## 🔴 Critique — le serveur tombe entièrement (ou se corrompt en mémoire)

### [x] 1. N'importe quelle erreur réseau tuait **tout le process**, pas juste le client fautif

**Corrigé, toujours vrai après le merge.** Un simple reset TCP (RST, voir
repro §4) tuait tout le process (`throw std::runtime_error("500")` dans
`handleRecv`/`handleSend`/`readCgiPipe`, catché seulement au niveau de
`main()` au lieu du client concerné). Vérifié encore fonctionnel dans
`WebServ::clientRead`/`handleRecv` post-merge : le `catch` est maintenant
bien localisé par connexion (`responseError`), pas de fuite jusqu'à `main`.

---

### [x] 2. Ctrl+C (SIGINT) ne marchait pas

**Corrigé, toujours vrai après le merge.** À l'origine, `signal(SIGINT, ...)`
n'était enregistré que dans un constructeur `Server` jamais utilisé par le
parsing réel (code mort) — Ctrl+C ne faisait rien, il fallait `kill -9`.
Maintenant dans `WebServ::WebServ(const char*)` (`WebServ.cpp:59`), appelé
une seule fois à la construction du runtime, indépendamment de quel
constructeur `Server` a servi à parser chaque bloc de config — plus robuste
que l'ancien design. Handler : `WebServ::handle_sigint` (`WebServ.cpp:65`).

---

### [x] 3. `pollfd.revents` non initialisé sur le pollfd des **listeners** — confirmé par Valgrind

**Corrigé pour ce site précis.** Dans l'ancien `Server::handleCgi()`, un
`pollfd pfd;` local n'initialisait jamais `.revents`, lu ensuite par erreur
dans la même itération de boucle avant que `poll()` n'ait eu la main. Le
site des listeners (`Server.cpp:393` à l'époque, `WebServ::run()`
aujourd'hui) fait bien `pfd.revents = 0;` et reste correct.

**⚠️ Mais le même bug de classe est revenu ailleurs après le merge — voir
#13 ci-dessous : le `pollfd` du pipe CGI, lui, n'a toujours pas ce
`revents = 0`.**

---

### [x] 12. 🆕 Use-after-free sur `Client` après `closeClient()` dans `WebServ::clientRead`

**Trouvé le 12/07 par Valgrind, pas dans la revue du 11/07.** Le plus grave
des bugs actuellement ouverts : facilement atteignable (n'importe quelle
connexion qui se ferme), risque réel de crash/corruption en soutenance.

`WebServ.cpp`, fonction `clientRead` :

```cpp
void WebServ::clientRead(size_t &i, int fd) {
    Client &client = _clients[fd];       // référence dans la map
    bool isHere = client.handleRecv();
    if (!isHere)
        closeClient(i, fd);              // erase(fd) -> détruit l'objet Client !
    else if (client.getStatus() == TRASH) { ... }
    else if (client.getStatus() == WRITTING) { ... }

    // <-- exécuté MEME quand isHere était false, sur `client` qui vient
    //     d'être détruit par closeClient() juste au-dessus
    if (client.getCounterTrash() <= 0 && client.getStatus() == TRASH) { ... }
}
```

Valgrind : `Invalid read of size 8` (`Client::getCounterTrash`) et
`Invalid read of size 4` (`Client::getStatus`), sur une adresse "112/176
bytes inside a block ... freed" par `WebServ::closeClient` → `_clients.erase`.
Sous un autre malloc (ASan, ou juste "par malchance" en prod) ce n'est plus
un warning Valgrind mais un crash ou une corruption silencieuse.

**Piste :** `return;` juste après le `closeClient(i, fd);` du `if (!isHere)`
(ou restructurer en early-return), pour ne plus jamais retoucher `client`
une fois le close effectué.

## 🟠 Sévère

### [x] 4. `_counter_trash` débordait (unsigned underflow) → le 413 ne partait jamais

**Corrigé, toujours vrai après le merge** (re-vérifié dans `Client.cpp`).
`_counter_trash` est un `size_t` ; `_counter_trash -= n` sans clamp
bouclait vers `SIZE_MAX` quand `n` dépassait ce qu'il restait à purger, et
`<= 0` sur un `size_t` ne redevenait jamais vrai après l'overflow — le 413
ne se construisait jamais, la connexion restait bloquée pour toujours.
Pendant le merge un cast signé supplémentaire (`static_cast<long>(...) - n`,
clampé à `0`) a été ajouté dans `handleRecv()` pour la même classe de
problème (voir aussi le fix de build noté plus bas).

### [x] 5. Le body d'une requête chunked n'atteignait jamais le script CGI

**Corrigé, toujours vrai après le merge.** Deux bugs empilés à l'origine
dans `Request::parseRequest()` côté chunked : la clé écrite était
`"Content-length"` (petit `l`) au lieu de `"Content-Length"` (map sensible à
la casse, jamais retrouvée par `parseCgi_env()` → `CONTENT_LENGTH=0` côté
CGI), et la valeur était affectée via `headers["Content-Length"] =
body.size();` — un `size_t` assigné à un `std::string`, qui compilait quand
même via une conversion implicite en `char` unique tronqué. Maintenant dans
`src/http/Request.cpp`, la bonne clé est utilisée avec une vraie conversion
via `std::ostringstream`.

### [x] 6. `close(pipe_body[2])` — accès hors-limites dans le fork CGI

**Corrigé, toujours vrai après le merge.** `pipe_body` est un `int[2]` ;
`pipe_body[2]` fermait un fd hors tableau (comportement indéfini) si le
second `pipe()` échouait. Non retrouvé dans le `Cgi.cpp` actuel.

### [x] 11. Dépassement potentiel de `_max_body_size` via un `Content-Length` énorme

**Corrigé, toujours vrai après le merge.** `strToInt()` (`utils/utils.cpp`)
fait `stringstream >> size_t` ; un `Content-Length` volontairement énorme
pouvait donner un résultat surprenant une fois casté en `long`. Pas
reproduit à l'exécution à l'époque. Pendant le merge, un `static_cast<long>`
erroné a été introduit puis corrigé côté `Client::isRequestCompleted` (les
deux opérandes de la comparaison sont bien des `size_t` maintenant, cast
inutile retiré) — voir "Build" plus bas.

## 🟡 Moyen — architecture / robustesse

### [ ] 7. Le routing multi-serveur (virtual host) ignore complètement le `Host` / le listener touché

**Toujours cassé, confirmé par les tests du 12/07** (5 échecs sur
`eval_tests.sh`, voir plus bas). `webserv.conf` a deux blocs `server` qui
partagent le port `8110` (`localhost` et `tchoutchou`), avec des
redirections `/old` différentes (`/files` vs `/new`). Peu importe le
`Host:` envoyé, ou même le port unique visé, c'est toujours la config du
1ᵉʳ serveur qui répond.

**Avant le merge :** `main.cpp` ne lançait que
`servers_vector[0].run(listeners)`.
**Après le merge :** même symptôme, cause légèrement différente —
`WebServ::handleReq`/`responseError`/`acceptNewClient` codent en dur
`_servers[0]`, avec 3 commentaires `TODO(discuss)` explicites dans le code
qui le confirment. Un test supplémentaire au 12/07 (port *unique* `:8130`,
sans ambiguïté de `Host` possible) confirme que c'est bien `_servers[0]` en
dur, pas juste un problème de matching de `Host`.

**Piste :** lier `listen_fd → Listener*` (savoir sur quel listener/port la
connexion est arrivée), puis `Listener → quels Server*` candidats (déjà
présent via `getLinkedServers()`), puis matcher le header `Host` parmi ces
candidats.

### [x] 8. Écriture du body CGI 100% bloquante

**Corrigé le 12/07, branche `fix/cgi-nonblocking-body-write`.** Le
`write()` unique et bloquant dans `Cgi::dadaExec()` a été remplacé par un
chemin non-bloquant piloté par `poll()`, symétrique à la lecture du pipe de
sortie :
- `Cgi::ParentExec` (renommée depuis `dadaExec`, `Cgi.cpp:113-127`) ne fait
  plus le `write()` : elle stocke le body dans `Client::_buffer_cgi_write`
  et garde `pipe_body[1]` ouvert (`Client::setCgiPipeFdWrite`).
- `WebServ::handleCgi` (`WebServ.cpp:159-170`) enregistre ce fd en
  `POLLOUT` dans `_poll_fds`, avec une nouvelle map `_pipe_to_client_write`
  (miroir de `_pipe_to_client`).
- `Client::handleSendCgi()` (`Client.cpp`, calqué sur `handleSend()`) fait
  l'écriture partielle avec offset ; `WebServ::writeCgiPipe` la pilote et
  ferme/`erase` le fd une fois le buffer épuisé.
- `loopPollFds` (`WebServ.cpp:274-277`) route vers `writeCgiPipe` pour les
  fds de `_pipe_to_client_write`.

**Vérifié en conditions réelles** (`webserv.conf`, port 8090) :
POST `/cgi-bin/serve.py` avec un body ~100 Ko (largement au-dessus de la
capacité d'un pipe, ~64 Ko sous Linux) → `201 Created` en ~70ms, aucun
hang, aucun process CGI/serveur qui traîne après coup. Avant fix : timeout
complet côté client (`HTTP_CODE:000`).

**Bugs rencontrés et corrigés en cours de route** (à titre de mémoire, pas
de suivi séparé nécessaire) : buffer d'écriture jamais rempli au premier
essai (`appendToBufferCgiWrite` déclarée mais jamais appelée), `pollfd.fd`
assigné à la mauvaise variable locale (`pfd` au lieu de `pfd_w`) laissant un
fd indéterminé enregistré dans `_poll_fds`, `send()` utilisé sur un fd de
pipe au lieu de `write()`, et `_offset_cgi_write` non initialisé dans les
constructeurs `Client` (soustraction `size_t` sous-underflow possible).

**Non traité, hors scope de ce fix** : pas de `signal(SIGPIPE, SIG_IGN)`
dans le code — si le script CGI ferme son stdin ou meurt avant d'avoir tout
lu, `write()` peut lever `SIGPIPE` et tuer le process serveur entier avant
que le code d'erreur de `handleSendCgi` ait la main. À tester (script qui
`exit()` sans lire stdin) et probablement à corriger séparément.

### [x] 9. `_pipe_to_client` grossit indéfiniment (fuite mémoire lente) — et #14, sa conséquence concrète

**Corrigé le 12/07.** `WebServ::readCgiPipe` (`WebServ.cpp:90-113`) fait
maintenant `_pipe_to_client.erase(fd)` et `closeClient(i, fd)`
**inconditionnellement** sur EOF (ligne 110-111), plus seulement dans le cas
`pid > 0` — la map ne grossit plus, et le pipe ne reste plus enregistré à
l'infini dans `_poll_fds` même quand le `Client` associé est le "fantôme"
décrit dans #14 ci-dessous (ce qui aurait pu provoquer un `poll()`
signalant EOF en boucle sur ce fd, jamais fermé).

### [x] 14. 🆕 Conséquence concrète confirmée par Valgrind (12/07)

Si une entrée obsolète de `_pipe_to_client` pointe vers un `client_fd` qui n'existe
plus dans `_clients`, `_clients[client_fd]` (via `operator[]`) **crée
silencieusement un `Client` tout neuf**, dont `_pid` n'était initialisé dans
**aucun** constructeur (`Client.hpp:30`) — et ce pid garbage partait
directement dans `waitpid(client.getPid(), NULL, WNOHANG)` dans
`readCgiPipe`. Valgrind : `Syscall param wait4(pid) contains uninitialised
byte(s)`.

**Corrigé** : `_pid` est maintenant mis à `0` dans les 4 constructeurs
"from scratch" de `Client` (`Client.cpp:27, 36, 46, 56`), et `readCgiPipe`
garde l'appel `waitpid` derrière `if (client.getPid() > 0)`
(`WebServ.cpp:103-109`) — un vrai PID de `fork()` est toujours positif, donc
`0` est une sentinelle sûre tant qu'elle est gardée (contrairement à `-1`,
qui a un sens spécial pour `waitpid` : "n'importe quel enfant").

### [x] 17. 🆕 `Client` fantôme jamais nettoyé de `_clients` (résiduel de #14)

**Corrigé le 12/07, branche `fix/orphan-cgi-client-cleanup`.**
`WebServ::readCgiPipe` et `WebServ::writeCgiPipe` (`WebServ.cpp:90-131`)
font maintenant `_clients.find(client_fd)` au lieu de `_clients[client_fd]`
directement. Si l'entrée n'existe pas (map `_pipe_to_client(_write)`
obsolète), la fonction nettoie le pipe correspondant
(`.erase(fd)` sur la map + `closeClient(i, fd)`) et retourne sans jamais
toucher `_clients` — plus de `Client` fantôme créé par `operator[]`.

Vérifié : `GET`/`POST` nominaux toujours `200`/`201` après le changement
(le `find()` ne coûte qu'une recherche de map de plus sur le chemin
normal, où l'entrée existe toujours).

### [ ] 10. `Client::~Client()` ne ferme aucun fd — tout repose sur `closeClient()` explicite

**Partiellement corrigé.** `Client::~Client(void) {}` (`src/client/Client.cpp:60`)
est toujours vide — ni `_poll_listen.fd` ni `_cgi_pipe_fd` ne sont jamais
fermés par le destructeur, tout repose sur le bon appel de `closeClient()`
partout (et on vient de voir avec #12 qu'un chemin l'oublie encore).

**Ce qui a changé pour le mieux :** au 11/07, aucun fd client accepté
n'était jamais passé en `O_NONBLOCK`. C'est maintenant corrigé dans
`WebServ::acceptNewClient` (`WebServ.cpp:120`,
`fcntl(client_fd, F_SETFL, O_NONBLOCK | FD_CLOEXEC)`), et le pipe CGI en
lecture l'est aussi (`WebServ.cpp:144`).

**Tentative du 12/07, revertée le jour même.** Un essai direct — fermer
`_poll_listen.fd`/`_cgi_pipe_fd`/`_cgi_pipe_fd_write` dans le destructeur —
casse **toutes** les connexions, pas seulement CGI : `Client` n'a ni
constructeur de copie ni `operator=` personnalisés (`Client.cpp:79` délègue
toujours à l'assignation générée par le compilateur, copie membre à membre
des fds). `WebServ::acceptNewClient` fait
`_clients[client_fd] = Client(client_fd, Request(), maxsize);` — le
`Client` temporaire construit `_poll_listen.fd = client_fd`, est copié dans
la map, puis détruit en fin d'expression. Avec un destructeur qui ferme les
fds, cette destruction ferme le fd du client **vivant** juste accepté.
Reproduit : `curl http://.../` → `Recv failure: Connection reset by peer`
immédiatement après connexion. Reverté avant merge.

**Piste correcte pour la prochaine tentative :** en C++98 (pas de move
semantics), il faut soit un vrai `operator=`/copy-ctor qui transfère la
propriété du fd (et invalide la source, ex: `src._poll_listen.fd = -1`
après copie), soit repenser le stockage pour ne plus jamais copier un
`Client` vivant (ex: `std::map<int, Client*>` avec `new`/`delete`, ou
insertion directe sans passer par un temporaire assigné).

### [x] 13. 🆕 `pollfd.revents` non initialisé pour le pipe CGI (régression du fix du point #3)

**Trouvé le 12/07 par Valgrind.** `WebServ::handleCgi` (`WebServ.cpp:136-149`)
construit un `pollfd pfd;` sur la pile et pousse `pfd.fd`/`pfd.events`, mais
**jamais `pfd.revents`** :

```cpp
pollfd pfd;
pfd.fd = client.getCgiPipefd();
fcntl(pfd.fd, F_SETFL, O_NONBLOCK | FD_CLOEXEC);
pfd.events = POLLIN;
// pfd.revents jamais mis à 0
_poll_fds.push_back(pfd);
```

Ce `push_back` a lieu **pendant** l'itération courante de `loopPollFds()`
(appelé depuis `clientRead`, lui-même dans la boucle `for` sur
`_poll_fds`), donc la taille du vecteur grandit en cours de route et la
boucle atteint cette nouvelle entrée **dans le même passage**, avant que
`poll()` n'ait eu la chance d'écrire son vrai `revents`. Valgrind :
`Conditional jump or move depends on uninitialised value(s)` dans
`loopPollFds()`. Le fix du point #3 avait couvert le pollfd du client et
des listeners mais pas celui-ci. **Piste :** `pfd.revents = 0;` avant le
`push_back`.

### [x] 15. 🆕 CGI POST répond avec 0 octet (chunked *et* Content-Length classique)

**Trouvé le 12/07 via `eval_tests.sh`, reproduit à la main.** Un POST vers
`www/cgi-bin/serve.py`, chunked ou en `Content-Length` classique, renvoie 0
octet au client — alors que le log d'accès serveur enregistre bien un
`200`. Donc la requête est traitée et une réponse est construite côté
serveur, mais n'arrive jamais sur le socket client. Pas de fd qui traîne
après coup (`/proc/<pid>/fd` propre sur un essai isolé), donc pas *a priori*
une resurgence de #9 en un seul coup. Le GET vers le même endpoint CGI
répond bien 200 avec du contenu.

**Root cause non identifiée.** Piste : loguer la taille de `_buffer_send`
juste avant le premier `send()` pour ce client, pour voir si le problème
est "la réponse n'est jamais écrite dans `_buffer_send`" ou "elle y est mais
n'est jamais envoyée" (`WebServ::clientWrite`/`Client::handleSend`).

### [ ] 16. 🆕 Crash du parser de config sur fichier tronqué (SIGABRT)

**Trouvé le 12/07 via `tests/config_errors/run_error_tests.sh`.** Un `.conf`
qui se termine juste après un mot-clé `server` ou `location` (sans le reste
