## `tests/eval_tests.sh` — 36 passed, 6 failed, 2 skipped

Échecs, tous rattachés à des causes déjà connues :

- **5 échecs = point #7** (routing vhost) : `/old` renvoie toujours la
  location du server 1 (`/files`) quelle que soit la combinaison
  port/`Host` visée vers le server 2 (`/new`), y compris sur le port
  *unique* du server 2 (`:8130`) où il n'y a même plus d'ambiguïté possible —
  confirme que c'est bien `_servers[0]` en dur, pas juste un problème de
  matching de `Host`.
- **1 échec nouveau : `chunked POST to CGI` → réponse vide.** Reproduit à la
  main avec `nc` (voir ci-dessous) : **pas spécifique au chunked** — un POST
  `Content-Length` classique vers le même script CGI (`www/cgi-bin/serve.py`)
  renvoie aussi 0 octet au client, alors que le log d'accès du serveur
  enregistre bien un `200`. Donc la requête est traitée et une réponse est
  construite côté serveur, mais elle n'arrive jamais sur le socket client.
  Pas de fd qui traîne après coup (`/proc/<pid>/fd` propre), donc pas (a
  priori) une resurgence du point #9 sur un seul essai. **Root cause non
  identifiée** — à creuser du côté `WebServ::clientWrite`/`Client::handleSend`
  pour ce chemin CGI précisément (le GET vers le même endpoint, lui, répond
  bien 200 avec du contenu). Un bon point de départ : logguer la taille de
  `_buffer_send` juste avant le premier `send()` pour ce client, pour voir si
  le problème est "la réponse n'est jamais écrite dans `_buffer_send`" ou
  "elle y est mais n'est jamais envoyée".
- **2 skips** : `ab`/`siege` — environnement (ports TIME_WAIT épuisés côté
  loopback, `siege` non installé), pas un bug applicatif.

## `tests/config_errors/run_error_tests.sh` — 38/40 + 2 "crash probes"

38 cas gérés proprement. Les 2 "probes" **crashent réellement** (signal 6,
`std::vector::operator[]` hors-limites) :

- `37_server_truncated_eof.conf` (fichier qui se termine juste après `server`)
- `38_location_truncated_eof.conf` (idem après `location`)

Root cause déjà documentée dans
`tests/config_errors/PARSING_ERRORS_REPORT.md` (section "Known issues
found by source review", point 2) : `Parser::parseStateDirective` lit
`_tokens_vector[index+1]`/`[index+2]` sans vérifier les bornes. Un fichier de
config tronqué juste après un mot-clé `server`/`location` fait planter tout
le process au lieu de renvoyer une erreur de parsing propre. Le rapport liste
aussi 3 autres points mineurs déjà connus (scope error text, code mort
`ERRS_PARSER_SCOPE_OVERFLOW`, `;` manquant en milieu de fichier mal
diagnostiqué) — voir le fichier pour le détail, rien de nouveau introduit par
le merge.

## Ce qu'il reste à faire, dans l'ordre où je m'y attaquerais

1. **#7 routing vhost** — le seul qui casse *plusieurs* comportements visibles
   (5 tests) et qui est probablement noté à l'éval 42 ("Host-based server
   selection"). Piste déjà notée le 11/07 : lier `listen_fd → Listener*` puis
   matcher le header `Host` parmi les servers candidats du listener, au lieu
   de toujours `_servers[0]`.
2. **CGI POST qui répond vide** (nouveau) — bloque potentiellement toute
   soumission de formulaire vers un CGI, à isoler avant la soutenance.
3. **#9 fuite `_pipe_to_client`** — petit fix mécanique (`.erase(fd)` en plus
   du `close()`), bon rapport effort/impact.
4. **Crash parser sur config tronquée** — ajouter une vérification de borne
   dans `Parser::parseStateDirective` avant `_tokens_vector[index+1/2]`.
5. **#8 write bloquant CGI** et **#10 destructeur `Client` vide** — plus gros
   chantiers (poll()-driven write, RAII sur les fds), à garder pour la fin
   comme convenu.
