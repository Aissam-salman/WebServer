### 4. Valgrind (leaks + fds) — reproduction manuelle du batch utilisé

```sh
make debug                     # -g3 -O0, plus lisible sous Valgrind
valgrind --leak-check=full --track-origins=yes --track-fds=yes \
         --log-file=/tmp/valgrind_report.log ./WebServ webserv.conf &
sleep 2                        # laisser le temps de bind les 3 ports

# --- batch de requêtes, à rejouer tel quel ---
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8090/                  # GET statique -> 200
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8090/doesnotexist      # -> 404
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8090/files/            # autoindex -> 200
echo "hello valgrind" > /tmp/vg_upload.txt
curl -s -o /dev/null -w "%{http_code}\n" -F "file=@/tmp/vg_upload.txt" http://127.0.0.1:8090/uploads/   # -> 201
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8090/uploads/vg_upload.txt                     # -> 200
curl -s -o /dev/null -w "%{http_code}\n" -X DELETE http://127.0.0.1:8090/uploads/vg_upload.txt           # -> 204
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8090/cgi-bin/serve.py  # CGI GET -> 200
printf 'POST /cgi-bin/serve.py HTTP/1.1\r\nHost: tchoutchou\r\nContent-Type: text/plain\r\nTransfer-Encoding: chunked\r\n\r\n5\r\nhello\r\n0\r\n\r\n' \
  | nc -w 3 127.0.0.1 8090                                                      # CGI POST chunked -> répro du bug #14 (réponse vide)
head -c 11000000 /dev/zero | tr '\0' 'A' > /tmp/vg_big.bin
curl -s -o /dev/null -w "%{http_code}\n" -X POST --data-binary @/tmp/vg_big.bin http://127.0.0.1:8090/uploads/  # -> 413
curl -s -o /dev/null -w "%{http_code}\n" -X BREW http://127.0.0.1:8090/          # méthode inconnue -> 405
curl -s -o /dev/null -w "%{http_code}\n" -H "Host: nope.example" http://127.0.0.1:8090/old  # vhost inconnu -> 301

kill -INT $(pgrep -f "WebServ webserv.conf" | head -1)  # SIGINT = arrêt propre, indispensable pour voir le vrai bilan Valgrind
cat /tmp/valgrind_report.log   # chercher "LEAK SUMMARY", "FILE DESCRIPTORS", "Invalid read", "uninitialised"
```

Piège rencontré : `pkill -f "WebServ webserv.conf"` peut se tuer lui-même si
le pattern matche la ligne de commande du shell qui l'exécute — préférer
`pgrep` pour choper le vrai PID puis `kill <pid>` ciblé, ou attraper le PID
au lancement (`... &`; `echo $!`).

## 🆕 Nouveaux bugs trouvés par Valgrind (pas dans la revue du 11/07)

Run : build `make debug`, Valgrind memcheck, batch de requêtes variées
(GET statique, upload/DELETE, CGI GET, CGI POST chunked, 413, 404, 405, vhost
inconnu), arrêt propre par SIGINT.

**Bilan mémoire du process principal à la sortie : `All heap blocks were
freed -- no leaks are possible`** (1101 allocs / 1101 frees). Donc pas de
leak "classique" — mais Valgrind a trouvé deux bugs de **use-after-free /
uninitialised read** bien réels pendant l'exécution :

### [ ] 12. Use-after-free sur `Client` après `closeClient()` dans `WebServ::clientRead`

`WebServ.cpp` (fonction `clientRead`) :

```cpp
void WebServ::clientRead(size_t &i, int fd) {
    Client &client = _clients[fd];       // référence dans la map
    bool isHere = client.handleRecv();
    if (!isHere)
        closeClient(i, fd);              // erase(fd) -> détruit l'objet Client !
    else if (client.getStatus() == TRASH) { ... }
    else if (client.getStatus() == WRITTING) { ... }

    // <-- exécuté MEME quand isHere était false, sur `client` qui vient
    //     d'être détruit par closeClient() juste au-dessus
    if (client.getCounterTrash() <= 0 && client.getStatus() == TRASH) { ... }
}
```

Valgrind : `Invalid read of size 8` (`Client::getCounterTrash`) et
`Invalid read of size 4` (`Client::getStatus`), sur une adresse "112/176
bytes inside a block ... freed" par `WebServ::closeClient` → `_clients.erase`.
Se déclenche dès qu'un client ferme/reset sa connexion (n'importe quel `curl`,
navigateur qui coupe, ou simplement une requête terminée dont le socket se
ferme) — donc très facilement atteignable en soutenance, pas un cas exotique.
Sous un autre malloc (ASan, ou juste "par malchance" en prod) ce n'est plus
un warning Valgrind mais un crash ou une corruption silencieuse.

### [ ] 13. `pollfd.revents` non initialisé pour le pipe CGI (régression du point #3 de la revue du 11/07)

`WebServ::handleCgi` (`WebServ.cpp:136-149`) construit un `pollfd pfd;` sur
la pile et pousse `pfd.fd` / `pfd.events`, mais **jamais `pfd.revents`** :

```cpp
pollfd pfd;
pfd.fd = client.getCgiPipefd();
fcntl(pfd.fd, F_SETFL, O_NONBLOCK | FD_CLOEXEC);
pfd.events = POLLIN;
// pfd.revents jamais mis à 0
_poll_fds.push_back(pfd);
```

Comme ce `push_back` a lieu **pendant** l'itération courante de
`loopPollFds()` (appelé depuis `clientRead`, lui-même dans la boucle `for`
sur `_poll_fds`), la taille de `_poll_fds` grandit en cours de route et la
boucle atteint cette nouvelle entrée **dans le même passage**, avant que
`poll()` n'ait eu l'occasion d'écrire son vrai `revents` — Valgrind : `Conditional
jump or move depends on uninitialised value(s)` dans `loopPollFds()`,
créé par la stack allocation dans `handleCgi`. Le fix vu le 11/07 pour ce
même problème (point #3) avait couvert le pollfd du client mais pas celui-ci.

### [ ] 14. Conséquence concrète du point #9 : `waitpid()` appelé avec un pid non-initialisé

`WebServ::readCgiPipe` : `int client_fd = _pipe_to_client[fd]; Client &client
= _clients[client_fd]; ... waitpid(client.getPid(), NULL, WNOHANG);`.
Valgrind : `Syscall param wait4(pid) contains uninitialised byte(s)`,
remontant à une "stack allocation" dans `map<int,Client>::operator[]`.
Ça confirme noir sur blanc le risque du point #9 : si `_pipe_to_client`
contient une entrée obsolète (jamais `erase()`é) pointant vers un `client_fd`
qui n'existe plus dans `_clients`, `_clients[client_fd]` **crée
silencieusement** un `Client` tout neuf (via `operator[]`), dont `_pid` n'est
initialisé dans **aucun** constructeur (`Client.hpp:30`, jamais dans une
init-list) — et ce pid garbage part directement dans `waitpid()`. Fixer #9
(faire le `.erase(fd)` manquant) résout aussi celui-ci ; initialiser `_pid`
dans les ctors serait une deuxième ligne de défense peu coûteuse.

## k6 — smoke test réduit (`VUS=10 ITERATIONS=150`, au lieu de 20/2000 par défaut)

Le serveur **survit** (pas de crash, pas de zombie, RSS stable 4.3→4.5 Mo,
fds stables à 6 = 3 listeners + stdio avant/après), donc pas de fuite ni de
blocage global détecté sur ce run réduit. Mais le taux d'échec est notable
pour seulement 10 VUs concurrents :

- `checks_failed`: 4.54% (8/176), `http_req_failed`: 29.33% (44/150).
- 2 `POST /uploads` (body normal, 1 Ko) ont carrément **timeout** côté k6
  (`request timeout`) — inattendu pour un si petit body.
- 4 `POST /uploads` avec body surdimensionné (55 Mo) n'ont **pas** reçu le
  413 attendu (mais pas de crash serveur non plus).
- `http_req_duration max=30s` alors que la médiane est à 224µs — quelques
  requêtes trainent visiblement beaucoup plus longtemps que les autres sous
  charge concurrente.

Pas eu le temps de root-causer précisément (capacité loopback avec des
bodies de 55 Mo × 10 VUs simultanés, vs. vrai bug de contention côté
serveur) — mais vu que #8 (write bloquant vers le pipe CGI) et la boucle
`poll()` mono-thread sont déjà identifiés comme des points de contention
potentiels, ce serait la première piste à éliminer avant de creuser plus
loin. À relancer avec les paramètres par défaut (`VUS=20 ITERATIONS=2000`)
une fois #7/#8/#12 traités, pour voir si le taux d'échec baisse.

## `tests/eval_tests.sh` — 36 passed, 6 failed, 2 skipped

Échecs, tous rattachés à des causes déjà connues :

- **5 échecs = point #7** (routing vhost) : `/old` renvoie toujours la
  location du server 1 (`/files`) quelle que soit la combinaison
  port/`Host` visée vers le server 2 (`/new`), y compris sur le port
  *unique* du server 2 (`:8130`) où il n'y a même plus d'ambiguïté possible —
  confirme que c'est bien `_servers[0]` en dur, pas juste un problème de
  matching de `Host`.
- **1 échec nouveau : `chunked POST to CGI` → réponse vide.** Reproduit à la
  main avec `nc` (voir ci-dessous) : **pas spécifique au chunked** — un POST
  `Content-Length` classique vers le même script CGI (`www/cgi-bin/serve.py`)
  renvoie aussi 0 octet au client, alors que le log d'accès du serveur
  enregistre bien un `200`. Donc la requête est traitée et une réponse est
  construite côté serveur, mais elle n'arrive jamais sur le socket client.
  Pas de fd qui traîne après coup (`/proc/<pid>/fd` propre), donc pas (a
  priori) une resurgence du point #9 sur un seul essai. **Root cause non
  identifiée** — à creuser du côté `WebServ::clientWrite`/`Client::handleSend`
  pour ce chemin CGI précisément (le GET vers le même endpoint, lui, répond
  bien 200 avec du contenu). Un bon point de départ : logguer la taille de
  `_buffer_send` juste avant le premier `send()` pour ce client, pour voir si
  le problème est "la réponse n'est jamais écrite dans `_buffer_send`" ou
  "elle y est mais n'est jamais envoyée".
- **2 skips** : `ab`/`siege` — environnement (ports TIME_WAIT épuisés côté
  loopback, `siege` non installé), pas un bug applicatif.

diagnostiqué) — voir le fichier pour le détail, rien de nouveau introduit par

## Ce qu'il reste à faire, dans l'ordre où je m'y attaquerais

1. **#12 use-after-free sur `Client` dans `clientRead`** — le plus grave :
   facilement atteignable (n'importe quelle connexion qui se ferme), risque
   réel de crash/corruption en soutenance. Fix mécanique : `return;` juste
   après le `closeClient(i, fd);` du `if (!isHere)`.
2. **#7 routing vhost** — casse *plusieurs* comportements visibles
   (5 tests) et probablement noté à l'éval 42 ("Host-based server
   selection"). Piste déjà notée le 11/07 : lier `listen_fd → Listener*` puis
   matcher le header `Host` parmi les servers candidats du listener, au lieu
   de toujours `_servers[0]`.
3. **CGI POST qui répond vide** (nouveau) — bloque potentiellement toute
   soumission de formulaire vers un CGI, à isoler avant la soutenance.
4. **#9 fuite `_pipe_to_client`** (+ son effet de bord #14, `waitpid` sur pid
   garbage) — petit fix mécanique (`.erase(fd)` en plus du `close()`), bon
   rapport effort/impact.
5. **#13 `pollfd.revents` non init pour le pipe CGI** — une ligne
   (`pfd.revents = 0;`) dans `handleCgi`.
6. **Crash parser sur config tronquée** — ajouter une vérification de borne
   dans `Parser::parseStateDirective` avant `_tokens_vector[index+1/2]`.
7. **#8 write bloquant CGI** et **#10 destructeur `Client` vide** — plus gros
   chantiers (poll()-driven write, RAII sur les fds), à garder pour la fin
   comme convenu.
8. **k6 : taux d'échec sous charge concurrente** (29% de `http_req_failed`
   sur un smoke test à seulement 10 VUs) — à re-tester une fois 1-5 corrigés,
   avant de creuser plus loin si ça persiste.
