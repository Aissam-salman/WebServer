## [ ] 7. Routing multi-serveur (`Host`) ignoré

Deux `server` peuvent partager un port dans `webserv.conf` — c'est toujours
le premier (`_servers[0]`) qui répond, quel que soit le `Host:` envoyé ou
même le port visé. En dur dans `WebServ::handleReq`/`responseError`/
`acceptNewClient` (3 `TODO(discuss)` dans le code le confirment).

**Piste :** `listen_fd → Listener*` (déjà tracé), puis `Listener →
Server*` candidats via `getLinkedServers()`, puis matcher le header `Host`
parmi ces candidats.

## [ ] 10. `Client::~Client()` vide — tout repose sur `closeClient()` explicite

Le destructeur ne ferme aucun fd. Tentative directe du 12/07 revertée : sans
copy-ctor/`operator=` custom (C++98, pas de move semantics),
`WebServ::acceptNewClient` copie un `Client` temporaire dans `_clients`, et
un destructeur qui ferme les fds détruit alors le fd du client **vivant**
au moment même de l'assignation (`_clients[fd] = Client(...)`) — casse
toutes les connexions (`Recv failure: Connection reset by peer`).

**Piste :** soit un vrai `operator=`/copy-ctor qui transfère la propriété
du fd (invalider la source après copie), soit ne plus jamais copier un
`Client` vivant (`std::map<int, Client*>`, ou insertion sans temporaire
assigné).

## [ ] 16. Crash du parser sur `.conf` tronqué (SIGABRT)

Un fichier qui se termine juste après `server`/`location` (bloc incomplet)
fait planter le process (signal 6, `std::vector::operator[]` hors-bornes)
au lieu de renvoyer une erreur de parsing propre.

**Cause :** `Parser::parseStateDirective` lit `_tokens_vector[index+1]`/
`[index+2]` sans vérifier les bornes. Détail complet dans
`tests/config_errors/PARSING_ERRORS_REPORT.md`.
